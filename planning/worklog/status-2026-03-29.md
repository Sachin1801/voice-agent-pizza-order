# Voice Agent Build Status — 2026-03-29

## What Has Been Built

### Phase 0: Foundation ✓
- **`src/config.ts`** — Zod-validated env loading. Reads all provider keys, server config, and feature flags. Fails fast with clear error messages on missing vars.
- **`src/types.ts`** — All TypeScript types: `OrderRequest` (matches spec payload exactly), `CallResult`, `CallOutcome`, `LLMAction` (discriminated union for 7 action types), `CallPhase`, `IVRState`. All validated with Zod schemas.
- **`tsconfig.json`** — Strict TypeScript config targeting ES2022/commonjs.
- **`package.json`** — Scripts for `dev`, `build`, `test`, `test:watch`.

### Phase 1: Observability ✓
- **`src/logging/event-schema.ts`** — Canonical event type with 14 required fields: `timestamp`, `call_id`, `session_id`, `sequence`, `component`, `event`, `level`, `direction`, `provider`, `correlation_id`, `latency_ms`, `state`, `message`, `data`, `redaction_level`. Stable enum-like component and event names.
- **`src/logging/logger.ts`** — Dual-output logger. Console: colorized, human-readable. File: structured JSONL for machine parsing. Auto-incrementing sequence numbers shared across child loggers. Event listeners for artifact writer integration.
- **`src/logging/artifact-writer.ts`** — Creates `data/runs/{call_id}/` per call with 7 files: `request.json`, `events.jsonl`, `transcript.jsonl`, `actions.jsonl`, `summary.md`, `result.json`, `metrics.json`. Optional `audio/` dir behind env flag.
- **`src/logging/redaction.ts`** — Masks sensitive keys (API keys, names, phones, addresses). Deep-redact for nested objects.
- **`src/logging/summary-writer.ts`** — Generates `summary.md` with outcome, timeline table, warnings/errors section, latency highlights. References event sequence numbers.
- **`src/logging/index.ts`** — Barrel export.

### Phase 2: Telephony ✓
- **`src/server.ts`** — Express + WebSocket server. HTTP routes mounted, WebSocket upgrade handler for `/api/twilio/media-stream`. Health check at `/health`.
- **`src/routes/calls.ts`** — `POST /api/calls` validates order with Zod, creates call via session manager, returns `{ call_id, status }`. `GET /api/calls/:call_id` returns status and result.
- **`src/routes/twilio-webhooks.ts`** — `POST /api/twilio/voice` returns TwiML that opens a bidirectional media stream WebSocket. `POST /api/twilio/status` receives Twilio status callbacks.
- **`src/session/call-session-manager.ts`** — Creates Twilio outbound calls, manages session state, handles WebSocket lifecycle. Has `sendDTMF()` (via Twilio REST) and `sendAudio()` (via WebSocket) methods.

### Phase 3: IVR State Machine ✓
- **`src/ivr/ivr-prompts.ts`** — 6 prompt configs (WELCOME→NAME→CALLBACK_NUMBER→ZIP_CODE→CONFIRMATION→TRANSFER) with regex patterns, response types (dtmf/speech/silence), and value derivation functions. Extracts zip code from address.
- **`src/ivr/ivr-state-machine.ts`** — Deterministic state machine. Pattern-matches STT transcripts against expected prompts. Handles retries (up to 3), IVR exhaustion hangup, and transition to HOLD. No LLM involvement.

### Phase 4: Hold Detector ✓
- **`src/conversation/hold-detector.ts`** — Monitors transcripts during hold. Distinguishes human greetings from hold music/automated messages using pattern matching. Requires multiple speech segments for medium-confidence detection. 5-minute default max hold timeout.

### Phase 5: Conversation Engine ✓
- **`src/audio/deepgram-client.ts`** — Streaming STT via Deepgram SDK v5. Accepts raw mulaw/8000. Emits interim + final transcript events. Uses `client.listen.v1.connect()`.
- **`src/conversation/prompts.ts`** — System prompt with full order context, substitution rules, behavior guidelines. Instructs Groq to return structured JSON actions only. Includes conversation context builder with order state. Versioned (`v1.0.0`).
- **`src/conversation/action-validator.ts`** — Parses Groq responses, extracts JSON (handles markdown wrapping), validates against `LLMActionSchema` with Zod. Returns typed `LLMAction` or error.
- **`src/conversation/rule-engine.ts`** — Code-level guardrail. Blocks no-go toppings even if LLM proposes them. Validates substitutions against acceptable list. Guards completion (requires pizza confirmed + special instructions delivered). Budget tracking and drink skip logic.
- **`src/conversation/conversation-engine.ts`** — Orchestrates LLM conversation. Maintains conversation history, order state. Calls Groq → validates → rule engine → speak or correct. Manual state update methods for prices, sides, drinks, delivery time, order number.

### Phase 6: Audio Bridge & TTS ✓
- **`src/audio/cartesia-client.ts`** — WebSocket TTS client. Sends text, receives mulaw/8000 audio chunks. Streams back for Twilio. Uses `pcm_mulaw` output format.
- **`src/audio/audio-bridge.ts`** — Bidirectional pipeline. Receives Twilio audio → forwards to Deepgram. Receives Cartesia TTS output → sends to Twilio. Barge-in support via `clearPlayback()`. No transcoding.

### Phase 7: Result Assembly ✓
- **`src/session/result-assembler.ts`** — Builds `CallResult` JSON from order state. Generates pizza description with substitutions applied. Writes all closing artifacts (result, metrics, summary).

### Phase 8: CLI & Test Fixtures ✓
- **`scripts/call.ts`** — CLI: `npx ts-node scripts/call.ts [order.json] [target_number]`. POSTs to API, polls for result, pretty-prints outcome.
- **`scripts/test-order.json`** — Sample order matching the spec (Jordan Mitchell, large thin pepperoni/mushroom/green pepper, buffalo wings, 2L Coke, $45 budget).

## Pipeline Wiring ✓ (Completed)

**The full pipeline is now wired in `call-session-manager.ts`.** All components are instantiated, audio is routed, transcripts drive phase transitions, and actions are executed.

### What flows end-to-end:

```
Twilio media event (base64 mulaw)
  → AudioBridge.processIncomingAudio() → Deepgram STT
  → routeTranscript() dispatches by phase:
    - IVR: IVRStateMachine.processTranscript() → executeIVRAction()
        → send_dtmf: Twilio REST call update
        → speak: Cartesia TTS → Twilio WebSocket
        → transition_to_hold: creates HoldDetector, switches phase
    - Hold: HoldDetector.processTranscript()
        → on human detected: creates ConversationEngine, switches to conversation
    - Conversation: ConversationEngine.addEmployeeSpeech() → generateResponse()
        → Groq LLM → ActionValidator → RuleEngine → speak via AudioBridge
        → hangup_with_outcome: writes artifacts, ends call
```

### Phase transitions:
- `initializing` → `ivr` (when audio pipeline connects)
- `ivr` → `hold` (when IVR state machine hits TRANSFER/HOLD)
- `hold` → `conversation` (when HoldDetector confirms human pickup)
- `conversation` → `completed` (on hangup action or call end)
